# keel

**Controlled execution governance for AI coding agents.**

Keel gives AI agents (Claude Code, Codex, or any code agent) a structured, auditable way to build software — one phase at a time — with gates, rollback, session tracking, and a full event ledger.

---

## Why

AI coding agents are capable but unconstrained. Without guardrails they tend to:

- Implement multiple features at once and lose scope
- Skip verification steps and write untested code
- Leave no record of what was built, when, or why
- Make it hard to roll back when something goes wrong

Keel solves this by wrapping the agent in a harness: a set of rules, commands, gate files, and audit records that the agent reads before touching any code. The agent works inside the harness, not around it.

**The result:** reproducible builds, phase-by-phase accountability, and a ledger you can query after the fact.

---

## How It Works

```
Your idea (PRD)
     │
     ▼
keel init          ← installs harness into your project
     │
     ▼
/keel-run 0       ← agent reads PRD, optionally generates TDD,
     │               generates BUILD_MANIFEST + phase build files
     ▼
/keel-run 1       ← agent implements phase, runs tests,
/keel-run 2         writes gate + rollback DAG, emits events
/keel-run 3
     │
     ▼
keel report session ← query what happened: tokens, tools, phases
keel rollback 3     ← undo a phase and everything downstream
```

Each phase is:
1. Governed by a `docs/phases/phase_N.md` file the agent reads
2. Verified by an exit-criteria checklist before the gate is written
3. Recorded in an append-only `events.jsonl` ledger
4. Snapshotted with a rollback DAG so it can be undone

## Install

### Homebrew (macOS / Linux)

```bash
brew tap locustave/keel
brew install keel
```

### From source (requires Go 1.25+)

```bash
git clone https://github.com/locustave/keel
cd keel
go build -o bin/keel ./cmd/keel
sudo mv bin/keel /usr/local/bin/keel
```

```bash
keel --help
```

## Quickstart

### 1. Initialize keel in your project

```bash
cd your-project
keel init
```

This creates:

```
your-project/
├── keel/                        # harness commands and rules
│   ├── commands/
│   │   ├── keel-run.md         # /keel-run slash command
│   │   ├── new-feature.md       # /new-feature slash command
│   │   ├── approve-feature.md   # /approve-feature slash command
│   │   ├── rollback-phase.md    # /rollback-phase slash command
│   │   └── verify.md            # /verify slash command
│   ├── hooks/                   # preflight, post-phase, and audit hooks
│   ├── rules/                   # agent governance rules
│   └── scripts/                 # preflight_context.py, verify scripts
├── docs/
│   ├── TDD.template.md          # TDD scaffold (filled during Phase 0)
│   └── phases/                  # generated by /keel-run 0
├── agent-rules.md               # repo constitution — agent reads this first
└── .agent/                      # gates, sessions, audit log, snapshots
```

### 2. Add your idea

Write `docs/PRD.md` describing what you want to build. That's all you need to start.

`docs/TDD.md` is optional — if missing, `/keel-run 0` will ask you five tech stack questions and generate it.

### 3. Open your AI agent and run Phase 0

In Claude Code, Codex, or any agent that reads slash commands:

```
/keel-run 0
```

Phase 0 will:
- Read your PRD (and generate the TDD if missing)
- Confirm the tech stack and write it to `agent-rules.md`
- Generate `BUILD_MANIFEST.yaml` with all product phases
- Generate a `docs/phases/phase_N.md` for every phase
- Write the Phase 0 gate file

### 4. Build phase by phase

```
/keel-run 1
/keel-run 2
/keel-run 3
```

Each `/keel-run N` call runs exactly one phase. The agent will not exceed the scope defined in the phase build file.

## CLI Reference

### `keel init`

Bootstrap the harness into a project.

```bash
keel init                             # current directory
keel init --repo /path/to/project
keel init --force-template            # overwrite existing template files
keel init --tracking off              # disable session tracking
```

### `keel current-phase`

Report the current phase state (which phase is active, last passed gate).

```bash
keel current-phase
keel current-phase --repo /path/to/project
```

### `keel verify`

Check harness consistency: required files, gate sequence, config validity.

```bash
keel verify
keel verify --phase 3                 # also verify a specific phase gate
```

### `keel plan-phases`

Derive a build manifest from the `## Deliverables` section of `docs/TDD.md`.

```bash
keel plan-phases                      # dry run — print proposed phases
keel plan-phases --confirm            # write BUILD_MANIFEST.yaml
keel plan-phases --tdd docs/TDD.md --start-phase 5
keel plan-phases --feature-slug my-feature   # write feature manifest
```

### `keel phase`

Emit phase lifecycle events to the session ledger. Called by the agent via `keel-run.md` checklist steps.

```bash
keel phase start 3
keel phase complete 3
keel phase failed 3
```

### `keel rollback`

Roll back a phase and all downstream phases that have rollback DAGs.

```bash
keel rollback 3                       # dry run — print what would happen
keel rollback 3 --confirm             # execute the rollback
keel rollback 3 --repo /path/to/project
```

Rollback:
- Deletes files created by the phase
- Invalidates the gate file (renames to `phase_N.rolled_back.json`)
- Repeats for all higher-numbered phases
- Appends a `rollback.executed` audit event
- Prints the `git reset --hard <sha>` command to run manually

#### `keel rollback write-dag`

Derive and write a rollback DAG from an existing gate file. Run this at the end of every phase, or retroactively for phases completed before rollback support was added.

```bash
keel rollback write-dag --phase 3
keel rollback write-dag --phase 3 --repo /path/to/project
```

Writes `.agent/snapshots/phase_3.rollback.json`.

### `keel session`

Manage tracking sessions. Sessions record every phase event, tool call, token usage, and debug loop in `.agent/sessions/<id>/events.jsonl`.

```bash
keel session start
keel session current
keel session list
keel session end
```

### `keel report`

Query the session event ledger.

```bash
keel report session                   # full session summary
keel report phase 3                   # report for a single phase
keel report tokens                    # token usage summary
keel report tokens --group-by phase   # tokens by phase
keel report tokens --group-by implementer
keel report tools                     # tool call counts and failure rates
keel report debug                     # debug loop count
```

### `keel merge-feature`

Merge a feature's phase plan into the project `BUILD_MANIFEST.yaml`. Normally called by `/approve-feature`.

```bash
keel merge-feature --slug my-feature
keel merge-feature --slug my-feature --dry-run
```

### `keel event`

Low-level command used by hook scripts to append events to the ledger.

```bash
keel event append --type tool_completed --tool-name Edit --actor-type keel
```

## Slash Commands (Agent Commands)

These are markdown files installed into `keel/commands/` by `keel init`. Your AI agent reads them as slash commands.

| Command | Purpose |
|---------|---------|
| `/keel-run N` | Execute build phase N under harness governance |
| `/keel-run 0` | Bootstrap: generate TDD (if missing), BUILD_MANIFEST, all phase build files |
| `/new-feature "name"` | Draft a feature workspace under `docs/features/{slug}/` |
| `/approve-feature {slug}` | Merge feature phases into `BUILD_MANIFEST.yaml` |
| `/rollback-phase N` | Interactively roll back phase N |
| `/verify` | Run harness verification checks |

## Feature Workflow

To add a new feature to a project already in flight:

```
/new-feature "CSV Export"
```

The agent drafts `docs/features/csv-export/` with scoped PRD, TDD, BUILD_MANIFEST, and STATUS files. You review and edit.

```
/approve-feature csv-export
```

Phases are renumbered and merged into the project `BUILD_MANIFEST.yaml`. Then run normally:

```
/keel-run 8
/keel-run 9
```

## Project Layout (after `keel init`)

```
.agent/
├── audit.jsonl                  # canonical audit event log
├── keel/
│   ├── config.yaml              # tracking configuration
│   └── current-session          # pointer to active session ID
├── phase_gates/
│   ├── phase_0.gate.json        # passed gate (status: "passed")
│   ├── phase_1.gate.json
│   └── phase_2.rolled_back.json # invalidated by rollback
├── sessions/
│   └── <session-id>/
│       ├── session.json         # session metadata
│       └── events.jsonl         # append-only event ledger
└── snapshots/
    └── phase_N.rollback.json    # rollback DAG per phase

keel/
├── commands/                    # slash commands (read by agent)
├── hooks/                       # preflight, phase, audit hooks
├── rules/                       # agent governance rules
└── scripts/                     # python helper scripts

docs/
├── PRD.md                       # product requirements (you write this)
├── TDD.md                       # technical design (generated or written)
├── TDD.template.md              # TDD scaffold used by Phase 0
└── phases/
    ├── phase_0.md
    ├── phase_1.md
    └── ...

agent-rules.md                   # repo constitution — read by agent on every run
BUILD_MANIFEST.yaml              # ordered phase plan
```

## Audit Log

Every significant event is appended to `.agent/audit.jsonl` in a canonical schema:

```json
{"event_type": "preflight.completed", "timestamp_utc": "...", "metadata": {"status": "completed", "tools": {...}}}
{"event_type": "phase.passed",        "timestamp_utc": "...", "phase": 1, "metadata": {"name": "..."}}
{"event_type": "rollback.executed",   "timestamp_utc": "...", "metadata": {"phases_unwound": [3, 4]}}
{"event_type": "feature.approved",    "timestamp_utc": "...", "metadata": {"slug": "csv-export"}}
```

Session events (tool calls, token usage, debug loops) live in `.agent/sessions/<id>/events.jsonl` and are queryable via `keel report`.
